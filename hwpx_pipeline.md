# HWPX 문서 자동 생성 파이프라인 — 개발 진행 보고

2026-05-11 기준

---

## 1. 현재 한 줄 요약

기존 HWPX 양식 파일을 분석하여, 원본 자료를 양식 구조에 맞게 생성하고 조립하는 파이프라인을 구축 중입니다. 현재는 장(章) 구분 문서와 얕은 보고서형 문서의 기본 생성 경로를 구현했으며, 다음 단계로 다중 섹션/붙임 보존 안정화를 진행합니다.

---

## 2. HWPX 문서 생성이 단순하지 않은 이유

HWPX 양식은 단순한 "빈칸 채우기"가 아닙니다. 하나의 양식 파일 안에는 다음이 섞여 있습니다:

- **제목란** — 문서 제목, 날짜, 기관명 등 고정 슬롯
- **본문 계층** — 대제목(□) → 항목(○) → 세부사항(*) 같은 다단계 구조
- **기호/번호 체계** — □, ○, Ⅰ/Ⅱ/Ⅲ, 1/2/3 등 단계별 기호 규칙
- **표** — 데이터 표, 현황 표, 실적 표 등
- **붙임/첨부** — 원본 그대로 유지해야 하는 첨부 자료
- **다중 레이아웃 섹션** — 페이지 설정이 다른 구간 (가로/세로, 여백 등)

AI에게 "이 양식에 내용을 넣어라"라고만 하면, AI가 양식 구조를 무시하고 자기 방식으로 재구성하거나, 기호를 잘못 붙이거나, 붙임 자료를 덮어쓸 수 있습니다.

그래서 **코드가 양식의 규칙과 구조를 먼저 파악하고, AI는 그 규칙 안에서 내용만 생성하도록 역할을 나누는 파이프라인**을 만들고 있습니다.

## 3. 전체 파이프라인 흐름

아래는 양식 파일과 원본 자료가 입력되어 문서가 생성되기까지의 전체 흐름입니다. AI와 코드가 번갈아가며 각자의 역할을 수행합니다.

```
[양식 HWPX + 원본 자료 입력]
         ↓
  ① 코드: HWPX 파일에서 문단/표/섹션 구조 추출
         ↓
  ② AI:  각 문단이 어떤 역할인지 분석
         (제목, 본문 항목, 보충 설명, 각주, 표 영역 등)
         ↓
  ③ 코드: 문단 스타일, 기호 후보, 계층 관계 정리
         ↓
  ④ AI:  비슷한 역할의 문단을 그룹으로 묶고,
         제목/본문/표/붙임 후보를 판단
         ↓
  ⑤ 코드: 양식의 문법 규칙(어떤 역할 아래 어떤 역할이 올 수 있는지),
         기호 정책(□ 고정, 1/2/3 순서 등), 계층 규칙 추출 및 검증
         ↓
  ⑥ AI:  양식의 영역 계획 수립
         - 제목란 영역
         - 본문 영역 (장 구분 또는 얕은 보고서)
         - 표 영역
         - 붙임/첨부 영역
         ↓
  ⑦ 코드: 양식 유형에 따라 생성 경로 선택
         - 장(章) 구분 문서 → 장별로 나눠서 생성
         - 얕은 보고서형 → 전체를 한 번에 생성
         ↓
  ⑧ AI:  원본 자료를 읽고, 양식 규칙에 맞는 내용 생성
         (기호 없이 순수 내용만 작성)
         ↓
  ⑨ 코드: 생성된 내용의 계층/기호/부모-자식 관계를 검증하고,
         기호를 양식 규칙에 따라 자동 부착
         ↓
  ⑩ 코드: 원본 HWPX 양식 파일에 생성된 내용 삽입
         - 제목란: 제목, 날짜, 기관명 채움
         - 본문: 양식 서식을 유지하면서 내용 교체
         ↓
  ⑪ 코드: 표, 붙임, 고정 영역 등 원본 유지 대상에 보존 정책 적용
         (다중 섹션/붙임 보존 정책은 다음 단계에서 안정화 중)
         ↓
[생성된 HWPX 문서 출력]
```

### 왜 이렇게 단계를 나누는가

| 역할 | AI가 잘하는 것 | 코드가 잘하는 것 |
|------|-------------|--------------|
| 문단 의미 파악 | "이 문단은 대분류 제목이다" | - |
| 비슷한 역할 그룹핑 | "이 문단들은 같은 성격이다" | - |
| 내용 작성 | 원본 자료를 요약하여 양식에 맞게 작성 | - |
| 기호/번호 부착 | - | □, ○, Ⅰ 같은 기호를 규칙에 따라 정확히 배치 |
| 계층 검증 | - | "대제목 아래 항목이 와야 한다" 같은 문법 규칙 검증 |
| 서식 유지 | - | 원본 양식의 글꼴, 여백, 문단 모양 복제 |
| 표/붙임 보존 | - | 원본 양식의 표와 붙임 자료를 그대로 유지 |

핵심은 **AI가 문서를 자유롭게 쓰는 것이 아니라, 코드가 만든 양식 규칙 안에서 내용만 생성하도록 제어하는 구조**라는 점입니다.

## 4. AI에게 넘기는 정보 vs 코드가 맡는 정보

### AI에게 넘기는 정보

| 정보 | 목적 |
|------|------|
| 원본 자료 텍스트 | 양식에 채울 내용의 원본 |
| 양식의 문단 역할과 계층 | "대제목 → 항목 → 세부사항" 같은 구조 |
| 각 역할의 설명과 예시 | "이 역할은 짧은 한 줄 제목이다" 같은 가이드 |
| 장 구성 또는 섹션 흐름 | "개요 → 현황 → 추진 → 계획 순서로 작성하라" |
| 출력 형식(schema) | 어떤 형태로 답해야 하는지 |

### 코드가 맡는 정보

| 정보 | 목적 |
|------|------|
| HWPX 파일 파싱 | 양식의 물리적 구조 읽기 |
| 문단/표/섹션 추출 | 양식의 구성 요소 분류 |
| 기호 자동 부착 | □, ○, Ⅰ 등 양식 규칙에 맞게 기호 배치 |
| 번호 순서 계산 | 같은 단계 항목의 순서 번호 (1, 2, 3) |
| 계층 검증 | 부모-자식 관계가 양식 문법에 맞는지 확인 |
| 원본 서식 복제 | 양식의 글꼴/여백/문단 모양 유지 |
| 표/붙임/고정 영역 보존 | 생성 대상이 아닌 영역을 원본 그대로 유지 |
| 최종 HWPX 조립 | 생성된 내용을 양식 파일에 물리적으로 삽입 |

## 5. 현재까지 구현된 것

### 양식 분석

- 양식 HWPX 파일의 문단별 역할, 계층, 기호 정책을 자동 분석
- 분석 결과를 캐시에 저장하여 같은 양식 재사용 시 즉시 처리
- 양식의 문법 규칙 추출 (어떤 역할 아래 어떤 역할이 올 수 있는지)
- 양식 영역 구분 (제목란, 본문, 표, 붙임 등)

### 내용 생성

- **장(章) 구분 문서**: AI가 원본 자료를 읽고 장별 제목을 결정한 뒤, 장마다 양식 규칙에 맞는 내용 생성
- **얕은 보고서형 문서**: 양식의 섹션 흐름을 파악하여 AI에게 전달하고, 그 흐름 안에서 내용 생성

### 기호/서식 처리

- AI가 순수 내용만 생성하고, 기호(□, ○, Ⅰ 등)는 코드가 양식 규칙에 따라 자동 부착
- 생성 결과의 계층 구조와 문법 규칙 준수를 코드가 자동 검증
- 문제 발생 시 원인을 추적할 수 있는 단계별 검증 로그 기록

### 문서 조립

- 생성된 내용을 원본 양식의 문단/기호/표/섹션 구조를 최대한 유지하면서 삽입
- 제목란(문서 제목, 날짜, 기관명) 자동 채움

### 설계 원칙

- 특정 양식에만 맞춘 규칙을 코드에 고정하지 않고, 양식 분석 결과를 기반으로 새 양식에도 확장 가능한 구조로 설계
- 현재 3개 대표 양식으로 검증 중이며, 추가 양식 확장 시 검증은 계속 필요

| 양식 | 유형 | 상태 |
|------|------|------|

| 주간 AI 활용 현황 (CC7) | 얕은 보고서형 | 양식의 4단계 흐름(개요→현황→추진→계획) 보존 확인 |

## 7. 최근 해결한 문제 — 양식 흐름 보존

### 문제

얕은 보고서형 양식(CC7)에서 AI에게 내용 생성을 맡기면, AI가 **원본 자료의 주제별로 문서를 다시 구성**하려는 경향이 있었습니다. 예를 들어, 양식이 "개요 → 현황 → 추진상황 → 향후계획" 순서인데, AI가 원본 자료에 3개 주제가 있으면 주제별로 3번 반복하는 구조를 만들었습니다.

### 개선

양식에서 관측된 섹션 흐름(어떤 대제목이 어떤 순서로 나오는지)을 코드가 추출하여 AI에게 전달했습니다. AI는 원본 자료를 주제별로 나누는 대신, **양식의 보고서 흐름 안에 내용을 배치**하도록 개선했습니다.

### 결과

- 양식의 "개요 → 현황 → 추진상황 → 향후계획" 흐름이 보존됨
- 원본 자료에 특화된 날짜나 주제명은 자연스럽게 조정됨
  - 예: 양식 원본의 "주간 AI 활용 현황 (3.30.~4.5.)" → "주간 현황 및 지표"

## 8. 현재 진행 중인 문제 — 영역별 처리 정책

### 문제

한 HWPX 파일 안에는 AI가 새로 작성할 본문 영역뿐 아니라, **원본 그대로 유지해야 하는 영역**(붙임 자료, 고정 서식, 표 등)이 함께 있습니다.

현재 장 구분 문서에서는 본문 영역의 생성에 집중하고 있어, 붙임/첨부 영역이 포함된 HWP 섹션에서 원본 유지 대상이 충분히 보존되지 않아 해당 섹션의 내용이 비어버릴 수 있는 문제가 관측되었습니다.

### 해결 방향

양식의 모든 영역을 순회하면서 **영역별 처리 정책**을 명시적으로 부여하는 구조를 만들고 있습니다:

| 처리 정책 | 의미 | 예시 |
|----------|------|------|
| **생성** | AI가 새 내용 작성 | 본문 각 장 |
| **값 채움** | 제목, 날짜 등 슬롯에 값 배정 | 제목란 |
| **원본 유지** | 원본 내용을 그대로 보존 | 붙임 자료, 첨부 서식 |
| **표 채우기 보류** | 표 구조는 유지하되 데이터 채움은 다음 단계에서 | 현황 표, 실적 표 |
| **건너뜀 (사유 기록)** | 처리하지 않되 이유를 기록 | 원본 자료가 부족한 영역 |

이렇게 하면 "AI가 새로 쓸 영역"과 "원본을 지킬 영역"이 명확히 구분되어, 본문 생성 과정에서 붙임 자료나 표가 비어버리는 문제를 방지할 수 있습니다.

## 9. 앞으로 남은 작업

| 순서 | 작업 | 설명 |
|------|------|------|
| 1 | **영역별 처리 정책 안정화** | 본문, 제목란, 붙임, 표, 고정 영역을 생성/보존/보류로 명확히 구분 |
| 2 | **다중 섹션/붙임 보존** | 붙임 자료가 포함된 섹션이 비어버리지 않도록 처리 |
| 3 | **원본 자료 배분 개선** | 원본 자료를 각 장/섹션에 더 정확하게 분배 |
| 4 | **표 셀 채우기** | 표 구조 보존을 넘어, 원본 자료의 데이터를 표 안에 반영 |
| 5 | **지식 베이스 연동** | 파일 직접 업로드 대신, 저장된 자료에서 관련 내용을 자동 선택 |
| 6 | **생성 근거 추적** | 생성된 문장이 원본 자료의 어느 부분을 기반으로 했는지 추적 |
| 7 | **내부 AI 전환 대비** | 현재 외부 AI API를 사용하지만, 내부 AI 서버로 전환해도 유지되도록 검증/규칙 구조 유지 |

## 10. 현재 진행 중인 문제와 해결 계획

| 항목 | 현재 상태 | 영향 | 해결 계획 |
|------|----------|------|----------|
| **붙임/첨부 섹션 내용 보존** | 본문 생성 과정에서 원본 유지 대상이 함께 제거될 수 있음 | 원본 유지가 필요한 섹션이 비어 문서가 불완전해질 수 있음 | 영역별 처리 정책으로 생성/보존 대상을 분리 |
| **원본 자료 장별 배분** | AI가 장 제목을 결정한 뒤 원본 자료를 나누는데, 한 장에 자료가 몰리거나 빈 장이 생길 수 있음 | 일부 장의 내용이 빈약해질 수 있음 | 배분 방식 재설계 예정 |
| **표 셀 데이터 반영** | 표는 현재 구조만 보존하고, 원본 자료의 데이터를 표 안에 채우지 않음 | 표가 핵심인 양식에서는 한계가 있음 | 표 셀 채우기 별도 개발 예정 |
| **다중 섹션 문서의 역할 구분** | 본문/붙임/고정 영역이 함께 있는 다중 섹션 문서에서 각 섹션의 역할을 구분하는 정책이 필요 | 섹션별 보존/배치 기준이 명확해야 올바른 문서 구조 유지 | 영역별 처리 정책과 함께 개선 예정 |
| **역할별 대표 서식 기준 조립** | 현재는 역할별 대표 서식을 기준으로 조립하며, 세부 서식 차이는 후속 개선 대상 | 일부 미세한 서식 차이는 추가 조정이 필요할 수 있음 | 서식 세분화 후속 개선 |
| **양식별 문체 반영 고도화** | 기본 구조와 역할 정보는 반영하고 있으며, 양식별 문장 길이/표현 밀도는 추가 개선 대상 | 양식별 표현 톤을 더 정교하게 맞출 여지가 있음 | 문체 프로필 고도화 검토 예정 |

## 11. 현재 제한 사항

- 표는 아직 새 데이터 자동 입력보다 구조 보존 중심으로 처리
- 원본 자료를 장별로 나누는 방식은 개선 예정
- 원본 양식의 문체/문장 밀도 반영은 추가 개선 대상
- 현재는 대표 3개 양식을 기준으로 검증 중이며 양식 확장 예정
- 생성된 내용의 출처 추적은 후속 단계에서 강화 예정
